System Architecture & Design Summary

This document details the design choices for our custom real-time audio transport protocol over UDP, built to survive hostile network conditions (drop, jitter, duplication, reordering) without exceeding strict bandwidth overhead limits.

1. The Wire Protocol

To support dynamic error correction and sequence tracking, we designed a custom 166-byte wire format (adding 6 bytes of overhead to the 160-byte payload):

  • uint8_t type: Identifies if the packet is Data (0) or FEC Parity (1).
  • uint32_t seq: The sequence number (or Base Sequence for parity packets).
  • uint8_t fec_size: The current redundancy block size.
  • uint8_t payload[160]: The actual audio payload or the computed XOR parity.

2. Receiver-Driven Intelligence

Because the receiver is the only component that actually knows the condition of the network, the architecture offloads almost all decision-making to the receiver. The sender simply packages data, calculates parity, and blindly listens for feedback.

Adaptive Forward Error Correction (FEC)

ARQ (retransmission) is fatal in real-time audio because round-trip times often exceed the playback deadline. Instead, we use proactive XOR-based Forward Error Correction. The sender groups packets and sends an XOR parity block. If one packet in that group is dropped by the relay, the receiver mathematically reconstructs it before the deadline using Missing = Parity XOR (Sum of Arrived).

3. The Feedback Loop

Every 500ms, the receiver calculates the exact packet loss percentage. It sends a 1-byte control message back to the sender via the feedback port (47003 -> 47004). If the network is clean, the receiver requests a large FEC group (e.g., 10 packets), minimizing bandwidth overhead. If burst loss is detected, the receiver instructs the sender to shrink the group size (down to 2), dramatically increasing protection while staying beneath the 2.0x overhead cap.

4. Strict Deadline Jitter Buffering

Rather than using a generic queue, the receiver relies on the system clock and strict math based on T0 and DELAY_MS. The receiver loop uses select() with a microsecond timeout calculated precisely against the absolute playback deadline of the next expected frame. This ensures we wait exactly as long as legally possible for late or out-of-order packets to arrive (or be recovered via FEC) without ever missing a deadline.

5. Pipeline Defenses

  • Deduplication: A hash set (std::unordered_set) of seen sequence numbers immediately drops duplicates introduced by the relay.
  • Reordering: A std::map naturally reorders packets by sequence number as they arrive over the wire.
  • Memory Safety: A sliding window mechanism aggressively garbage-collects old packets and FEC blocks to prevent memory leaks in the endless loop.
